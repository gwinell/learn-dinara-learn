## Введение: что такое терминал

### Что это и зачем

Терминал (ещё его называют "консоль" или "командная строка") — это программа, в которую ты **набираешь текстовые команды**, а компьютер их выполняет. Вместо того чтобы кликать мышкой по папкам и кнопкам, ты **пишешь** что нужно сделать.

Представь: вместо того чтобы открыть папку кликом, ты пишешь «открой папку Desktop» — и компьютер открывает. Вот это и есть терминал.

### Как открыть терминал

**На Mac:**
1. Нажми `Cmd + Space` (это Spotlight — быстрый поиск)
2. Напиши `Terminal`
3. Нажми `Enter`
4. Откроется белое (или чёрное) окно с текстом — это и есть терминал

**На Linux (Ubuntu и подобные):**
1. Нажми `Ctrl + Alt + T`
2. Или найди в меню приложений "Терминал"

### Как выглядит терминал

После открытия ты увидишь что-то вроде:

```
gwinell@MacBook ~ %
```

Это называется **промпт** — приглашение к вводу. Всё что справа от `%` (или `$`) — это то, куда ты набираешь команды. Мигающий курсор показывает место, где появится текст.

### Базовые команды, которые пригодятся

Вот 4 команды, которые ты будешь использовать постоянно:

**`pwd`** — показать где ты сейчас (полный путь текущей папки)
```bash
pwd
# Вывод: /Users/gwinell/Documents/LearnDinaraLearn
```
Зачем: чтобы не потеряться. Как GPS для папок.

**`ls`** — показать содержимое текущей папки
```bash
ls
# Вывод: D01T01_Инструкция.md  D01T01.ID_1577481-1  D02T02_Инструкция.md ...
```
Зачем: чтобы посмотреть какие файлы и папки здесь лежат.

**`ls -la`** — то же, но подробнее (показывает скрытые файлы, права, размеры)
```bash
ls -la
# Вывод: drwxr-xr-x  5 gwinell  staff   160  ...  .git
#        -rw-r--r--  1 gwinell  staff  1234  ...  README.md
```
Зачем: часто нужно знать есть ли скрытые файлы (начинаются с точки) и какие у файлов права.

**`cd`** — перейти в другую папку (Change Directory)
```bash
cd Documents          # перейти в папку Documents (от текущей)
cd ..                 # перейти на уровень вверх (к родительской папке)
cd ~/Documents        # перейти в Documents из любого места (~ = домашняя папка)
cd /                  # перейти в корень диска
```
Зачем: чтобы перемещаться между папками. `cd` = "change directory" = "сменить папку".

> **ВАЖНО:** `cd ..` — это две точки. Две точки = "родительская папка". Одна точка = "текущая папка".

### Как копировать команды из этой инструкции

1. Выдели команду в инструкции мышкой
2. Нажми `Cmd+C` (Mac) или `Ctrl+C` (Linux) — скопировал
3. Кликни на окно терминала
4. Нажми `Cmd+V` (Mac) или `Ctrl+Shift+V` (Linux) — вставил
5. Нажми `Enter` — выполнил

> **ВАЖНО:** В терминале `Ctrl+C` — это **НЕ копирование**, а **остановка текущей команды**! На Mac копирование через `Cmd+C`. На Linux копирование через `Ctrl+Shift+C`.

### Как редактировать файлы в nano

nano — это простой текстовый редактор в терминале. Вот как им пользоваться:

```bash
nano мой_файл.txt      # открыть файл в nano
```

После открытия:
- **Стрелки** — двигать курсор
- **Просто печатай** — текст вводится туда где курсор
- **Backspace / Delete** — удалять символы

Внизу экрана есть подсказки:
- `^O Write Out` — значит нажать `Ctrl+O` чтобы **сохранить**
- `^X Exit` — значит нажать `Ctrl+X` чтобы **выйти**

**Порядок действий:**
1. Открыл файл: `nano мой_файл.txt`
2. Сделал изменения (напечатал/удалил текст)
3. Сохранил: нажал `Ctrl+O`, потом `Enter` (подтверждение имени файла)
4. Вышел: нажал `Ctrl+X`

> **ВАЖНО:** Не закрывай терминал без `Ctrl+O` + `Ctrl+X` — изменения не сохранятся!

## Подготовка: Git workflow

Прежде чем начать квесты, нужно сделать пару подготовительных шагов.

### Шаг 1: Клонировать проект

Git — это система контроля версий. Она хранит историю изменений твоих файлов. GitLab — это веб-сайт, где хранятся Git-репозитории (проекты).

**Клонирование** = скачивание проекта с GitLab к себе на компьютер.

```bash
# Перейти в папку, куда хочешь скачать проект
cd ~/Documents/LearnDinaraLearn

# Скачать (склонировать) проект
git clone git@gitlab.com:your-school21/D01T01.ID_1577481.git

# Перейти внутрь скачанного проекта
cd D01T01.ID_1577481
```

Что делает каждая команда:
- `cd ~/Documents/LearnDinaraLearn` — перейти в папку, где лежат твои проекты
- `git clone ...` — скачивает проект с GitLab. `git@gitlab.com:...` — это адрес проекта (SSH-ссылка)
- `cd D01T01.ID_1577481` — зайти внутрь скачанного проекта

> **ВАЖНО:** Адрес `git@gitlab.com:your-school21/D01T01.ID_1577481.git` — это пример. У тебя будет свой адрес. Найди его на платформе School 21 (кнопка "Clone" → SSH).

### Шаг 2: Создать ветку develop

В Git **ветка** — это как параллельная версия проекта. Ты можешь менять файлы в одной ветке, не ломая другую.

По умолчанию ты на ветке `master` (или `main`). Нужно создать ветку `develop` — в ней ты будешь работать.

```bash
# Создать ветку develop и сразу переключиться на неё
git checkout -b develop
```

Что делает `git checkout -b develop`:
- `git checkout` — команда для переключения между ветками
- `-b` — флаг "создай новую ветку"
- `develop` — имя новой ветки

**Проверь что ветка создалась:**
```bash
git branch
```
Должен увидеть:
```
* develop
  master
```
Звёздочка `*` показывает какая ветка активна. Должна быть `develop`.

### Шаг 3: Как делать коммиты и пушить

**Коммит** — это "сохранение" в Git. Как Ctrl+S, но с комментарием что ты изменил.

**Пуш** — это отправка твоих сохранений на GitLab (в облако).

Вот полный цикл:
```bash
# 1. Добавить файлы в "очередь на коммит"
git add src/                # добавить папку src/
# или
git add .                   # добавить ВСЁ

# 2. Сделать коммит (сохранение)
git commit -m "feat: описание что ты сделал"

# 3. Отправить на GitLab
git push origin develop
```

Что делает каждая команда:
- `git add src/` — говорит Git "я хочу сохранить изменения в папке src". Без `git add` коммит не увидит изменения.
- `git commit -m "..."` — создаёт сохранение (коммит) с комментарием. Текст в кавычках — описание что ты сделал.
- `git push origin develop` — отправляет коммиты на GitLab. `origin` = "удалённый репозиторий" (GitLab), `develop` = какую ветку пушить.

> **ВАЖНО:** Делай `git push` после каждого квеста, чтобы ничего не потерялось.

### Полезные сокращения (алиасы)

Алиасы — это короткие имена для длинных команд. Настрой их один раз:

```bash
git config --global alias.st status      # git st = git status
git config --global alias.co checkout    # git co = git checkout
git config --global alias.br branch      # git br = git branch
git config --global alias.ci commit      # git ci = git commit
git config --global alias.lg "log --oneline --graph --all"  # git lg = красивая история коммитов
```

После этого можешь писать `git st` вместо `git status` и т.д.

### Совет: команда history

Команда `history` показывает все команды, которые ты набирал в терминале. Это пригодится когда нужно записать команды в `.sh` файл.

Если команд было много и ты хочешь сохранить только последние N строк:
```bash
history 30 > quest3.sh    # сохранить последние 30 команд
history 50 > quest3.sh    # сохранить последние 50 команд
```

Просто подбери число под ситуацию — сколько команд ты набрал для решения квеста.

```bash
# 1. Перейти в папку с проектами
cd ~/Documents/LearnDinaraLearn

# 4. Создать ветку develop

# 5. Проверить что всё получилось
```

```bash
git branch     # Должно показать * develop
```

```bash
# Шаг 1: Добавить все файлы
git add .

# Шаг 2: Сделать коммит
# ВАЖНО: пиши точно так, без лишних кавычек и символов!

# Шаг 3: Отправить на GitLab

```

> **ВАЖНО про кавычки в коммите:** Используй обычные двойные кавычки `"`. Не ставь лишних кавычек, не используй `«»` или `' '`. Просто скопируй команду как написано.

### Что могло пойти не так

**Ошибка: `Permission denied (publickey)`**
- Значит твой SSH-ключ не настроен на GitLab
- Решение: зайди на GitLab → Settings → SSH Keys → добавь свой ключ
- Как получить ключ: `cat ~/.ssh/id_rsa.pub` (скопируй вывод и вставь на GitLab)
- Если ключа нет: `ssh-keygen -t rsa -b 4096 -C "твой_email@example.com"` → потом `cat ~/.ssh/id_rsa.pub`

**Ошибка: `Repository not found`**
- Значит неверный адрес репозитория
- Проверь адрес: зайди на GitLab → проект → кнопка "Clone" → скопируй SSH-ссылку

**Ошибка: `fatal: destination path already exists`**
- Значит папка уже скачана
- Решение: либо удали папку `rm -rf D01T01.ID_1577481` и клонируй заново, либо просто зайди в неё `cd D01T01.ID_1577481`

**Ошибка: `fatal: not a git repository`**
- Значит ты не в папке проекта
- Проверь: `pwd` должен показывать путь к D01T01.ID_1577481
- Если нет: `cd D01T01.ID_1577481`

**Ошибка: `error: pathspec 'develop' did not match`**
- Значит ветка develop ещё не создана
- Сначала сделай: `git checkout -b develop`

**Ты написал `git commit` без флага `-m`**
- Откроется редактор коммитов (vim или nano)
- Не пугайся! Нажми `:wq` и `Enter` (в vim) или `Ctrl+X` → `Y` → `Enter` (в nano)
- Или нажми `Ctrl+C` чтобы отменить и начать заново с `-m`

## Quest 2. Repo

### Что нужно сделать

Запустить скрипт `src/ai_initial_module.sh` и исправить ошибки, из-за которых он падает. **Нельзя менять сам скрипт** — нужно исправить файлы, с которыми он работает.

### Что за скрипт

Это интерактивный скрипт. Он:
1. Спрашивает твоё имя
2. Пытается запустить другой скрипт `ai_module_2.sh`
3. Если всё хорошо — выводит "МОДУЛЬ 2 успешно загружен!"
4. Если нет — падает с ошибкой

### Проблемы (что не так)

Когда ты запустишь скрипт, он упадёт с ошибкой. Вот почему:

1. **Скрипт `ai_module_2.sh` не имеет прав на выполнение** — Linux не даёт запускать файлы, у которых нет "права на запуск". Это как файл в Windows с расширением .exe — без прав он не запустится.

2. **Файл `important_data_for_ai_module_2.txt` содержит неправильные данные** — в нём написано `1, 2, 42`, а скрипт ожидает ровно 5 строк с числами 1, 2, 3, 4, 5.

### Пошаговое решение

```bash
# Сначала попробуй запустить скрипт и посмотри ошибку
bash src/ai_initial_module.sh
# Увидишь ошибку — это нормально, сейчас мы её починим
# Нажми Enter если скрипт ждёт ввода, или Ctrl+C чтобы остановить

# Шаг 1: Дать скрипту ai_module_2.sh права на выполнение
chmod +x src/ai_module_2.sh
# Что делает: chmod = "change mode" (изменить режим), +x = добавить право на запуск (execute)
# Без этого команда ./ai_module_2.sh в скрипте упадёт с "Permission denied"

# Шаг 2: Исправить файл данных
# Нам нужно перезаписать файл правильными данными. Используем команду cat:
cat > src/important_data_for_ai_module_2.txt << 'EOF'
1
2
3
4
5
EOF
# Что делает: перезаписывает файл правильными данными
# Работает так:
#   cat > файл — "хочу записать что-то в этот файл"
#   << 'EOF' — "вот что я буду писать, пока не увижу строку EOF"
#   1\n2\n3\n4\n5 — сами данные (каждое число на новой строке)
#   EOF — "стоп, закончил"
#
# ВАЖНО: строка EOF в конце должна быть БЕЗ ПРОБЕЛОВ и ПЕРЕД НЕЙ НЕ ДОЛЖНО БЫТЬ ПРОБЕЛОВ!

# Шаг 3: Проверить что файл правильный
cat src/important_data_for_ai_module_2.txt
# Должно вывести:
# 1
# 2
# 3
# 4
# 5

# Шаг 4: Запустить скрипт снова
bash src/ai_initial_module.sh
# Он спросит имя — введи любое (например "Динара") и нажми Enter
# Должен вывести: "МОДУЛЬ 2 успешно загружен!"
```

### Коммит

```bash
# Добавить изменения
git add src/

# Сделать коммит
git commit -m "fix: исправлены права на скрипт и данные модулей"

# Отправить
git push origin develop
```

### Что могло пойти не так

**Ошибка: `Permission denied` при запуске скрипта**
- Значит ты забыл дать права: `chmod +x src/ai_module_2.sh`

**Скрипт всё равно падает**
- Проверь что файл данных правильный:
```bash
cat src/important_data_for_ai_module_2.txt
```
- Должно быть ровно 5 строк: 1, 2, 3, 4, 5
- Если там что-то другое — перезапиши (см. Шаг 2 выше)

**Ошибка: `No such file or directory`**
- Значит ты не в папке проекта
- Проверь: `pwd` должен показывать путь к D01T01.ID_1577481
- Если нет: `cd ~/Documents/LearnDinaraLearn/D01T01.ID_1577481`

**Команда `cat > файл << 'EOF'` не работает / выдаёт странную ошибку**
- Убедись что ты копируешь команду правильно, без лишних символов
- Проверь что после `<< 'EOF'` нет ничего лишнего
- Проверь что строка `EOF` в конце стоит на отдельной строке и без пробелов перед ней

**Ошибка: `git: 'add' is not a git command`**
- Убедись что написал `git add`, а не `git addd` или `git add .` (с точкой)
- Проверь что Git установлен: `git --version`

**Ты случайно написал `git add .` вместо `git add src/`**
- Это не страшно, `.` добавит ВСЁ, что тоже сработает
- Но лучше привыкнуть добавлять конкретные папки

## Quest 3. Dirs & Files

### Что нужно сделать

Это самый сложный квест. Нужно исправить структуру папок и файлов, чтобы скрипт `ai_door_management_module.sh` заработал.

Скрипт ищет файлы в определённых папках. Но из-за ошибки в структуре он их не находит. Наша задача — расставить всё как нужно.

### Что есть сейчас (неправильно)

```
src/
├── door_management_fi/          # НЕПРАВИЛЬНО: папка обрезана, должно быть door_management_files
│   ├── door_1.conf              # Все .conf файлы в одной папке (а надо в door_configuration/)
│   ├── door_2.conf
│   ├── ...
│   ├── door_16.conf
│   ├── door_21.conf             # Лишний файл (нужны только 1-16, плюс 17)
│   ├── door_1.log               # Все .log файлы тут (а надо в door_logs/)
│   ├── door_2.log
│   ├── ...
│   ├── door_16.log
│   └── door_map_1.1             # А надо в door_map/
├── ai_door_control.sh           # Нет прав на выполнение
└── ...
```

### Что должно быть (правильно)

```
src/
├── door_management_files/       # ПРАВИЛЬНО: полное имя
│   ├── door_configuration/      # Подпапка для .conf файлов
│   │   ├── door_1.conf
│   │   ├── door_2.conf
│   │   ├── ...
│   │   ├── door_16.conf
│   │   └── door_17.conf         # Добавлен (из-за бага в скрипте)
│   ├── door_logs/               # Подпапка для .log файлов
│   │   ├── door_1.log
│   │   ├── door_2.log
│   │   └── ...
│   │   └── door_16.log
│   └── door_map/                # Подпапка для карты дверей
│       └── door_map_1.1
├── ai_door_control.sh           # С правами на выполнение
└── ...
```

### Пошаговое решение

```bash
# Сначала перейди в папку src
cd src
# Что делает: теперь все команды работают относительно папки src

# === ШАГ 1: Посмотри что сейчас в папке ===
ls
# Должен увидеть: door_management_fi, ai_door_control.sh и другие файлы
# Это "До" — то что сейчас неправильно

# === ШАГ 2: Переименовать папку ===
mv door_management_fi door_management_files
# Что делает: mv = "move" (переместить/переименовать)
# Переименовывает door_management_fi → door_management_files

# Проверь что переименовалось:
ls
```

## Quest 7. Branches

### Что нужно сделать

Создать цепочку Git-веток по определённой схеме. Это самый сложный квест в плане понимания Git.

### Что такое ветки в Git (простым языком)

Представь что у тебя есть книга. Ветка — это как **закладка**. Ты можешь:
- Читать книгу с закладки A
- Поставить новую закладку B
- Писать заметки на закладке B, не затрагивая то что записано на закладке A
- Потом "слить" заметки с закладки B обратно в основную книгу

В Git:
- `master` (или `main`) — основная ветка ("книга")
- `develop` — рабочая ветка ("закладка для черновиков")
- `key-branch`, `feature/3-key`, `release/1.0` — дополнительные ветки

### Схема что нужно сделать

Вот что описано в файле `src/git_for_human/readme.txt`:

1. Работаем на ветке `develop`
2. Создаём `key-branch` от `develop` — копируем туда ключ
3. Делаем копию ключа `main-2.key` и коммитим
4. Создаём `feature/3-key` от `key-branch`
5. Делаем копию ключа `main-3.key` и коммитим
6. Сливаем `feature/3-key` в `key-branch`
7. Создаём `release/1.0` от `develop`
8. Сливаем `key-branch` → `develop` → `release/1.0`

### Пошаговое решение

```bash
# === ШАГ 1: Убедиться что ты на develop ===
git checkout develop
# Что делает: переключается на ветку develop
# Если ты уже на develop — ничего не произойдёт, это нормально

# Проверь что ты на develop:
git branch
# Звёздочка * должна быть рядом с develop

# === ШАГ 2: Создать key-branch ===
git checkout -b key-branch
# Что делает: создаёт новую ветку key-branch от develop и переключается на неё

# Проверь:
git branch
# Должно показать:
# * key-branch
#   develop
#   master

# === ШАГ 3: Скопировать ключ ===
cp src/ai_help/main.key src/ai_help/main-2.key
# Что делает: копирует main.key → main-2.key

# Проверь что файл появился:
ls src/ai_help/main-2.key
# Должен показать путь к файлу

# === ШАГ 4: Добавить файл в Git ===
```

> **ВАЖНО:** Если GitLab ругается что ветки не отслеживаются, используй:
> `git push -u origin key-branch` (для каждой новой ветки первый раз)
> Флаг `-u` связывает локальную ветку с удалённой.

### Что могло пойти не так

**Ошибка: `fatal: A branch named 'key-branch' already exists`**
- Значит ветка уже создана
- Проверь: `git branch`
- Если нужная ветка есть — переключись: `git checkout key-branch`

**Ошибка: `error: pathspec 'develop' did not match any file(s) known to git`**
- Значит ты на неправильной ветке или опечатался
- Проверь текущую ветку: `git branch`
- Проверь что пишешь `develop` правильно (без опечаток)

**Ошибка: `fatal: refusing to merge unrelated histories`**
- Это редкая ошибка в этом квесте
- Если появилась — `git merge --abort` и перечитай шаги

**При `git merge` конфликт**
- В этом квесте конфликтов быть не должно (мы просто копируем файлы)
- Если вдруг есть — `git merge --abort` и перечитай шаги

**`git log --oneline --graph --all` показывает не то что ожидалось**
- Проверь что все коммиты были сделаны (никакой команды не пропущено)
- Убедись что названия коммитов начинаются с имени ветки

**Забыл начать коммит с названия ветки**
- Если коммит ещё НЕ запушен — можно исправить:
```bash
git commit --amend -m "правильное сообщение с названием ветки"
```
- Если коммит УЖЕ запушен — лучше не трогать, просто делай следующий коммит правильно

**Ты написал(а) `git merge` вместо `git checkout` перед ним**
- Git выдаст ошибку или сделает не то что нужно
- Всегда сначала переключись на ветку КУДА сливаешь: `git checkout ветка_куда`
- Потом: `git merge ветка_откуда`

**Ты перепутал(а) порядок слияний**
- Схема: feature/3-key → key-branch → develop → release/1.0
- Если слил(а) не в ту сторону — `git merge --abort` и начни шаг заново

**Ошибка: `fatal: not a valid object name`**
- Значит опечатка в имени ветки
- Проверь имена: `git branch` покажет все ветки

**Ты случайно сделал(а) коммит не на той ветке**
- Не паникуй! Это можно исправить
- Способ 1: `git log --oneline -1` посмотри последний коммит
- Способ 2: если коммит не запушен — можно переместить через `git cherry-pick`
